Erlang 游戏服务器缓存设计

发表于 2025-11-05 更新于 2026-03-09 分类于 游戏开发

概述

游戏服务器是典型的数据密集型系统，对实时性要求高，缓存设计直接影响响应速度与磁盘 I/O。本文介绍四种 Erlang 游戏服务器缓存架构，并结合实践分析其优劣。

一、Erlang 常用缓存方式

方式 特点
mnesia 全服共享，支持分布式部署
ETS 内存表，全服共享
进程字典 进程独享数据
进程 state 进程状态数据

二、缓存设计一：玩家进程 + ETS + 定时读写

2.1 架构示意

2.2 运行机制

阶段 说明
服务器启动 功能业务从数据库加载数据，缓存到 ETS 或进程字典；业务维护修改标志；功能进程每 30 分钟持久化变更数据
玩家登录 加载玩家全部功能数据到玩家进程 state，同步 state 数据块到 ets_online；玩家功能维护 state 与修改标志；玩家进程每 30 分钟持久化变更；非玩家业务通过 ets_online 访问在线玩家数据
玩家下线 持久化玩家变更数据，删除 ets_online 中该玩家数据；非玩家业务直接访问数据库

2.3 优缺点

维度 说明
优点 内存占用小；完整玩家 state 数据块保证消息处理原子性
缺点 业务实现复杂，需维护数据修改与持久化逻辑；访问离线玩家数据 I/O 压力大；业务模块间因冗余玩家数据而耦合增加

三、缓存设计二：mnesia + 缓存管理进程 + 超时回收

3.1 架构示意

3.2 运行机制

阶段 说明
服务器启动 为每张数据表创建对应 mnesia 内存表；需常驻的表预加载；访问时未命中则从数据库加载到 mnesia 再返回，并为数据 key 添加超时时间戳交给缓存定时器管理；更新时与 mnesia 缓存对比（无则先加载），判断新增或更新后同步到数据库并更新超时时间戳；超时（如 1 小时）后，缓存管理进程从 mnesia 删除对应 key
玩家登录 玩家数据导入进程字典；玩家进程每 30 分钟与缓存对比并同步更新到数据库；非玩家业务从 mnesia 缓存访问，未命中则从表加载
玩家下线 需持久化的数据同步到缓存和数据库；访问玩家数据时走 mnesia 缓存（未命中则加载并加入超时定时器）

3.3 复合主键加载管理

  • 复合主键表在 mnesia 中建为 bag 类型
  • 加载时仅能用第一个主键加载该 key 下的全部数据
  • 写入时通过复合 key 对比判断新增或更新

3.4 优缺点

维度 说明
优点 业务开发简单，无需关心加载与持久化；超时机制可回收冷数据，降低内存
缺点 在线玩家缓存非实时，部分业务可能不满足

四、缓存设计三：mnesia + 表订阅进程

4.1 架构示意

4.2 运行机制

阶段 说明
服务器启动 为每张数据表创建 mnesia 内存表；每张表启动「修改订阅进程」和「持久化进程」；修改订阅进程定期（如 10 分钟）对数据更新做持久化；数据访问需先手动加载到 mnesia 再访问；数据更新写入 mnesia 后发送给表订阅进程，订阅进程常做数据更新合并
玩家登录 玩家数据加载到 mnesia 缓存和玩家进程字典；更新时同步更新进程字典与 mnesia，并将更新操作发送给订阅进程；通过加载 mnesia 缓存访问玩家数据

4.3 复合组件加载

  • 复合组件加载时，为每张表增加唯一 ID 字段作为主键，插入时生成新 ID

4.4 优缺点

维度 说明
优点 业务开发简单；无需关系数据加载与持久化；数据缓存与操作合并减少大量磁盘 I/O；表订阅进程可灵活调整缓存方式，提高项目适应性
缺点 数据加载后不回收，内存随运行时间增长

五、缓存设计四：表缓存进程 + 持久化进程

5.1 架构示意

5.2 运行机制

阶段 说明
服务器启动 为每张数据表启动缓存进程和持久化进程；常驻内存表数据预加载；数据访问通过 call 缓存进程；未命中则从数据库加载到缓存表；数据更新时 call 缓存进程，未命中则先加载再更新，并同步到持久化进程；持久化进程定期将变更写入数据库
玩家登录 直接与 mnesia 缓存数据交互
复合主键加载 加载时按第一个主键加载该 key 下全部数据；缓存进程维护该主键下已加载的数据 key 列表；后续访问可基于该列表查询

5.3 优缺点

维度 说明
优点 业务开发简单，无需手动处理关系数据加载与持久化；数据缓存与操作合并显著减少磁盘 I/O；缓存在数据中心核心服务内部署，不影响访问性能
缺点 数据加载后若变冷不会自动回收，长时间运行内存会增长

六、四种设计对比

设计 核心存储 持久化方式 适用场景
设计一 玩家 state + ETS 玩家/功能进程定时写库 在线玩家为主，对原子性要求高
设计二 mnesia 缓存管理 + 超时删除 需冷热分离、内存可控
设计三 mnesia + 表订阅 表订阅进程定时持久化 需灵活缓存策略、减少 I/O
设计四 表缓存进程 + mnesia 独立持久化进程 多表、需统一缓存与持久化

七、选型建议

  • 小规模、强一致性：设计一，state 原子性好
  • 中大规模、内存敏感：设计二，超时回收冷数据
  • 高 I/O 压力：设计三、四，合并缓存与持久化
  • 多表、统一管理：设计四，表级缓存进程清晰
