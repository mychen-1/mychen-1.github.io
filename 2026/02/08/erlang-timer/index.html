erlang定时器

发表于 2026-02-08 更新于 2026-03-03 分类于 Erlang

概述

本文简单介绍 Erlang 中常见的几种定时器用法，以及底层 erl_timer 时间轮的实现原理。重点讨论一个常见问题：当取消定时器时，超时消息已经到达邮箱，该如何处理？

1. 引言

前几天在开发群中，大家讨论了这样一个场景：调用 erlang:cancel_timer/1 取消定时器时，对应的超时消息其实已经投递到了进程邮箱，该怎么办？

大致有两种看法：

  1. 允许回调继续执行：既然回调消息已经到达邮箱，说明此时取消已经“晚了”，就让回调正常处理。
  2. 强制保证取消语义：对于某些业务，必须保证取消操作一定生效，否则副作用太大，需要额外处理。

个人观点是：结合业务场景来看待。
对需要精确控制的充值类业务，必须更严格地保证取消语义；而对于一些广播、通知类需求，可以适当放宽要求——多处理一次通常问题不大。

2. 消息已到达邮箱时如何取消？

1. Erlang 中几种定时器

  1. erlang:start_timer(Time, Pid, Msg): 在 Time 毫秒后向 Pid 发送一条 {timeout, Ref, Msg} 消息。
  2. erlang:send_after(Time, Pid, Msg): 与 start_timer/3 类似，只是回调消息格式不同，直接是 Msg。
  3. timer 模块：基于 gen_server + ETS 管理，底层还是调用上面两个接口。在定时器数量很多的情况下，需要注意其性能问题。

2. 解决方案：参考 gen_fsm 的 cancel_timer/1

一个现成的方案是参考 gen_fsm（后来是 gen_statem）里 cancel_timer/1 的实现思路：

  • 先调用 erlang:cancel_timer(Ref) 尝试取消；
  • 如果返回取消失败（false），说明有两种可能：
    • 定时器已经触发，超时消息已经在邮箱里但还未来得及处理；
    • 消息已经被业务代码消费掉。
  • 此时再用一次 receive 扫描邮箱，把还没被消费的 {timeout, Ref, Msg} 主动“捞”出来。

3. Erlang 定时器的底层实现

这一节主要参考：
https://blog.csdn.net/u012785877/article/details/72805375

该篇博客详细介绍了 C 层 time 模块如何实现一个高效的定时器系统，支持百万级别定时器。这里对该博客做一个简要总结，方便记忆整体结构。

数据结构：

通过数据结构可以大致看出设计理念：

结合 ErtsTWheelTimer *slots[1+ERT_TW_SOON_WHEEL_SIZE+ERTS_TW_LATER_WHTTL_SIZE] 三个参数可知，对应三个轮槽：

  1. 立即轮（at_once）
  2. 近槽轮（soon）
  3. 远槽轮（later）

erl_timer 中的 slot 即是 ErtsTimerWheel_ -> slots 的下标。

那对于不同的定时器，计算出来的 slot 可能一样，即 slot 冲突时是怎么解决的？

time 接口：

接口的详细介绍可以看上面引用的博客，这里只总结几个要点：

  1. 定时器的增删查

    1. 增：通过 erts_create_timer_wheel 初始化，再用 erts_twheel_set_timer 放入 ErtsTimerWheel 中，这一步会计算 slot，判断放入哪一个槽；槽内冲突通过 insert_timer_into_slot 链表挂接。
    2. 删：remove_timer 根据 slot 信息从对应槽中摘除。
    3. 查：find_next_timeout 查找“最近一次到期”的定时器时间点，配合 ErtsTimerWheel_ 中的 next_timeout_time、next_timeout_pos 字段和轮询逻辑使用。

  2. 轮询嘀嗒：erts_bump_timers
    负责驱动时间轮的推进，并触发到期定时器；同时做了限流（ERTS_TWHEEL_BUMP_YIELD_LIMIT），避免一次性处理太多定时器把调度器“占死”。

总体流程

  1. erl_timer_sup：管理进程，erts_init_time_sup 完成初始化。
  2. tick 轮询时，先比较 current_time 与 next_timeout_time，只有当前时间晚于最近到期时间才会进入轮询。
  3. time 层面进入 erts_bump_timers 做一次时间轮“嘀嗒”。
  4. 等待下一次 tick 继续上述流程。

设计小结

  1. 数据结构：采用“时间轮槽 + 链表”的结构，并进一步分为立即轮、近槽轮、远槽轮。查找槽位是 (O(1))，槽内遍历是 (O(n))，整体非常适合大规模定时器场景。
  2. 限流机制：bump_timers 每次只处理有限个定时器，避免长时间轮询导致其他调度任务饥饿。
  3. 快速跳过无效槽位：维护 next_timeout_time、next_timeout_pos，使得调度器可以快速判断是否有必要扫描时间轮，从而减少无意义的轮询。
  4. 超长定时器优化：对于超长时间定时器，保持在同一槽位，结合树结构等方式减少槽位占用，提高整体性能。
