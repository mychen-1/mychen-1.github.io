erlang节点-分布式

发表于 2026-02-08 更新于 2026-03-03 分类于 Erlang

概述

简单记录一下 Erlang 节点和分布式通信的几个核心概念：节点命名方式、EPMD、节点连接模式，以及心跳机制的大致实现思路。

1. 节点介绍

概念： 一个 Erlang 虚拟机实例，就是一个节点（node）。

1.1 节点命名方式

  1. 无命名节点：直接启动 erl，没有名字，不能参与分布式通信。
  2. 短名节点（short name）：erl -sname short
    形如 short@hostname，主机名不带完整域名。
  3. 长名节点（long name）：erl -name long@ip_or_fqdn
    形如 long@192.168.0.10 或 long@host.example.com。

注意：
节点之间互连时，短名只能连短名，长名只能连长名，不能短连长或长连短。

2. 如何连接节点

Erlang 底层通过 TCP 做节点间通信，由 EPMD 负责端口管理和节点发现。

2.1 EPMD 介绍

EPMD（Erlang Port Mapper Daemon）主要职责是管理 Erlang 节点监听端口：

  • 第一个 Erlang 分布式节点启动时，会自动拉起 EPMD。
  • EPMD 默认监听端口是 4369。
  • 以物理机为单位：同一台机器上可以有多个节点，但只会有一个 EPMD 进程。

2.2 连接方式

节点 A 要连节点 B，大致有两种方式找到对方：

  1. 通过节点名解析
    类似 DNS 解析，通过 short@hostname / long@fqdn 映射到 IP + 端口。
  2. 直接使用 IP
    手工指定 long@ip 的形式，由 EPMD 帮你找到目标节点端口。

3. 节点互连模式

每个节点都会维护自己“已连接节点”的信息。
连接模式分为两种：

  • normal（正常连接）
  • hidden（隐藏连接）

区别在于连接会不会继续向外扩散：

  • hidden：只在双方之间建立连接，不会把对方的其他连接关系继续扩散出去。
  • normal：会把双方已经连接的所有节点都互相打通（节点扩散）。

举个例子：三个节点 A、B、C。

  • hidden 模式： A -> B，B -> C
    • A 的连通信息：[A, B]
    • B 的连通信息：[A, B, C]
    • C 的连通信息：[B, C]
  • normal 模式： A -> B，B -> C
    • A 的连通信息：[A, B, C]
    • B 的连通信息：[A, B, C]
    • C 的连通信息：[A, B, C]

4. 心跳机制

一个常见问题：hidden 模式下还有没有心跳机制？

Erlang 中与节点网络相关的逻辑主要集中在 net_kernel 模块。

源码中可以看到一个 ticker_loop 接口（loop 循环），每次 tick 会向 kernel（实际上是 net_kernel 进程）发送 tick 消息。

这里有两个关键问题：

  1. ticker_loop 是什么时候开始循环的？
  2. 向 kernel 进程发 tick 到底是干什么？

4.1 ticker_loop 何时启动

在 net_kernel:init/1 中（大致 500 行附近），初始化 net_kernel 的时候，会专门 spawn 一个进程来执行 ticker_loop，负责定时发 tick。

4.2 tick 发给谁，做了什么

在 net_kernel 的 handle_info 里可以看到，会用到 State#state.conn_owners。

  • conn_owners：每有一个远程节点连上来，就会在里面新增一条 {Pid, NodeName} 的记录。
  • 也就是说，每个节点连接，对应一个负责处理该连接的进程 Pid1。

再往下看 inet_tcp_dist:setup，新连接一个节点时会走到这里，继续深入到 dist_util:con_loop：

  • 对每个节点连接都有一个 loop 进程（上面的 Pid1），负责收发该连接上的消息。
  • 这个进程会接收 {kernel, tick} 相关的消息。

在 con_loop 中可以看到，当收到 tick 时，会调用 send_tick：

  • send_tick 底层调用 send(Socket, [], [force]) 发送一个空包。
  • 这个空包就是应用层的心跳包。

4.3 小结：节点心跳整体流程

综合起来，可以得到这样的流程：

  1. 建连时，根据节点名或 IP，发起到对方机器的连接。
  2. 校验通过后，为每个远程节点 spawn 一个进程 Pid1，执行 con_loop。
  3. net_kernel 进程在 init 时启动一个 ticker_loop 循环。
  4. 每个 tick 周期，ticker_loop 向所有 Pid1 发送 tick 消息。
  5. 每个 Pid1 收到 tick 后，向对应远程节点发送一个空包心跳。

4.4 一些细节

  • 有一个 nextticktime 字段记录下一次 tick 时间，当双方节点之间有正常业务通信时，会重置 nextticktime，避免无意义心跳过多。
  • 为什么不用 TCP 自带的 keepalive？
    TCP 心跳属于传输层，默认间隔较长（资料中常见默认值是 2 小时），从应用层视角来看，感知断线太慢，所以 Erlang 自己在应用层实现了更频繁的心跳机制。
  • 两个节点 A、B，如果 A 给 B 发了心跳，那 B 还需要再给 A 发吗？
    需要。双方各自维护自己的心跳，互相确认“我还活着”。
