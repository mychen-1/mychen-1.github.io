erlang节点-分布式

发表于 2026-02-08 分类于 Erlang

概述

Erlang节点

1.节点介绍

概念：一个erlang 虚拟机实例

命名方法：

  1. 无命名 erl
  2. 短命名： erlang sname shot
  3. 长命名： erlang name long@ip

节点互连 只可以 短-> 短 | 长连长。

2.如何连接节点。

erlang底层采用TCP连接， EPMD 进行维护。

EPMD介绍： （Erlang Port Mapper Daemon）： 管理erlang端口进程

  • 第一个节点启动
  • 固定端口 4396
  • 物理机为单位， 一台机器上 多个节点也只有一个EPMD

连接方式：

1.对方命名。 有类似DNS解析， 可以通过命名转换 为 IP 端口。

2.Ip

3.节点互连模式。

每一个节点 都有自己已连接节点信息。

节点 分为 正常连接 和 hidden连接。

区别：节点扩散， hidden 只会扩展双方节点。 而正常连接会将 双方已经连接的所有节点都连接起来。 （节点扩散）

例子： A， B， C 三个节点。

​ hidden模式：A -> B， B -> C, 此时 各节点的连接信息 A： [A, B] . B : [A, B, C] C:[B, C]

​ 正常模式：A -> B， B -> C, 此时 各节点的连接信息 A： [A, B， C] . B : [A, B, C] C:[A, B, C]

4.心跳机制

hidden模式下， 会不会有心跳机制。

erlang 关于网络模块 在net_kernel.

%% todo ticker——loop接口图片

其中 我们看到一个 ticker_loop接口。 这里是一个 loop循环， Tick后 会向 Kernel（kernel其实是net_kernel进程） 发生tick消息。

有2个问题。

1.ticker_loop 什么时候开始循环

  1. 向 Kernel进程发 tick干什么

问题一：.ticker_loop 什么时候开始循环

%% todo init 接口

可以看init 接口， 589行， net_kernel初始化的时候 开启一个进程专门做tick

问题二：.向 Kernel进程发 tick干什么

%% todo handle——info接口 tick 图片

可以看到是使用 state.conn_owners.

介绍一下 conn_owners: 每有一个节点连接， 会新增信息进来。 {Pid => nodename}

%% todo 看state信息 手机上，

我们可以看到 Pid 第一个是0， 可以知道是本地进程， 那这个进程收到消息干了什么？

可以看 inet_tcp_dist:setup接口， 新连接一个节点会走到这里。 继续往里面走到 dist_util:con_loop接口。

发现面对每一个 节点连接都有一个进程执行loop， 进程就是 此PID， loop中会接受 {Kernel.tcik}消息。

其中会发生 send_tick 消息。 这个接口底层是调用 send(Socket,[], [force) 发生一个空包， 即心跳。

由此可知结论：

  1. 连接节点， 根据命名或者ip， 发送到对方机器。
  2. 校验通过后， 为每个节点 spwan一个进程 叫PID1， 执行conn_loop操作。
  3. net_kernel进程 记录 PID1进行， init的时候 启动一个 ticker_loop循环。
  4. 每个 Tick的时间， 向PID1发生tick消息。
  5. PID1接收到tick信息， 向对方节点发送心跳。

部分细节：

1.有一个 nextticktime维护， 记录下次tick间隔， 当双方节点有 通信时， 会重置 nextticktime

2、TCP也有心跳这些， 为什么不用？ TCP属于传输层， 断开感知太长， 到应用层可能断开很久了。 看资料可能2小时。

3.A B 2节点， A向B发送了心跳， 那B还要发给A吗？ 需要。 告诉活的很正常。
