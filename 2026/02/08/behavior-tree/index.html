行为树介绍

发表于 2026-02-08 更新于 2026-03-03 分类于 杂记

概述

行为树（Behavior Tree，简称 BT）是一种常见的 AI 决策结构，广泛用于游戏 NPC、机器人和自动化流程控制。
它把复杂行为拆成可组合的节点，每帧（或每次调度）从根节点开始执行，并最终返回一个状态。
本文只覆盖行为树的基础概念和常规落地流程，不展开进阶设计。

行为树是什么

行为树本质上是“树形状态机”，可以理解为比传统有限状态机更易组合和扩展的一种结构，由节点和执行器组成：

  • 执行器每次从根节点开始 tick
  • 节点根据规则向下执行子节点
  • 子节点返回状态，父节点据此决定下一步

行为树常见四种返回状态：

  • SUCCESS：当前节点成功完成
  • FAILURE：当前节点失败
  • RUNNING：当前节点还在执行，下次继续
  • ERROR：配置或运行异常（可选，部分实现不用）

节点分类

1) 组合节点（Composite）

用于组织子节点的执行顺序，是树的骨架。

  • Sequence（序列，类似 AND）
    从左到右执行，遇到 FAILURE 立即失败；全部成功才成功。
  • MemSequence（记忆序列）
    与 Sequence 类似，但会记住上次运行位置，适合长流程。
  • Priority / Selector（选择，类似 OR）
    从左到右尝试，遇到 SUCCESS 立即成功；都失败才失败。
  • MemPriority（记忆选择）
    记住上次命中的分支，减少重复判断开销。

2) 装饰器节点（Decorator）

用于“包裹”一个子节点并修改其行为。

  • Repeat：重复执行 N 次
  • RepeatUntilFailure：直到失败才停止
  • RepeatUntilSuccess：直到成功才停止
  • Inverter：反转结果（成功<->失败，运行态一般保持不变）
  • Limiter：限制最大执行次数

3) 条件节点（Condition）

只做判断，不执行动作。例如：

  • 是否有目标
  • 是否在攻击范围内
  • 生命值是否低于阈值

4) 动作节点（Action）

执行具体行为。例如：

  • 移动到目标点
  • 播放技能
  • 巡逻到下一个路径点

执行机制（open / tick / close）

在 behavior3 这一类实现中，节点一般有三个关键回调：

  • open：节点首次进入时触发（初始化）
  • tick：每次调度都会调用（核心逻辑）
  • close：节点结束时触发（清理状态）

典型顺序：

open -> tick -> (RUNNING 时反复 tick) -> close

如果节点持续返回 RUNNING，下一帧会继续进入该节点，直到返回成功或失败。

黑板（Blackboard）

黑板是行为树共享数据存储，常用于跨节点通信和在多次 tick 之间保留上下文：

  • 记录目标、路径、冷却时间
  • 保存临时状态（如“本次巡逻索引”）
  • 做上下文隔离（全局 / 树级 / 节点级）

实践中更推荐把“会变化的数据”放黑板，不要写死在节点内部。

behavior3 + Erlang 落地流程

1. 在 behavior3Editor 画树并导出 JSON

导出文件通常包含：

  • id：全局唯一 ID
  • name：节点名
  • category：节点类别（composite/decorator/action/condition）
  • properties：节点参数（次数、时间、阈值等）
  • children：子节点列表

2. 读取 JSON 并构建运行时结构

3. 缓存到 ETS（按你当前思路）

可拆成三类表（命名示例）：

  • bt_title：行为树元信息
  • bt_tree：树定义
  • bt_node：节点定义（节点 ID -> 配置）

4. 角色登录后绑定行为树

流程建议：

  1. 根据角色类型和权重选择树
  2. 初始化黑板（角色 ID、初始目标等）
  3. 定时 tick 执行
  4. 把执行结果回写到战斗/移动系统

行为树的优缺点

优点：

  • 可视化强，便于策划和程序协作
  • 模块化高，复用节点成本低
  • 易扩展，新增行为通常只改局部子树

缺点：

  • 树过大后维护成本会上升
  • 黑板数据混乱时排查困难
  • RUNNING 节点处理不当会出现“卡行为”

实战建议

  • 统一节点命名（动词开头），方便定位日志
  • 为关键节点加上调试日志：进入、退出、返回状态
  • 为 RUNNING 节点设置超时或兜底失败条件
  • 为黑板字段做命名空间设计，避免冲突
  • 将“条件判断”和“动作执行”分离，减少副作用
