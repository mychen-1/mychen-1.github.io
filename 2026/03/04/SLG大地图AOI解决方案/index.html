SLG 大地图中关于超长物体的 AOI 解决方案

发表于 2026-03-04 更新于 2026-03-05 分类于 游戏开发

概述

AOI（Area of Interest，兴趣区域）是游戏服务器中不可或缺的技术，用于管理玩家视野范围内的对象进入、离开、移动及状态变化等事件，是支撑战斗场景中玩家交互的基础。本文介绍常见 AOI 方案及其优劣，并重点讨论超长物体（如行军路径）的 AOI 同步问题及解决方案。

一、九宫格 AOI 基础

1.1 九宫格方法

九宫格是最常用的 AOI 技术：将地图划分为等大的格子，每个格子内包含各类对象。玩家的可见范围是以其视野中心所在格子为中心的 3×3 九宫格。

  • 客户端显示范围：通常略小于服务器传输范围
  • 优化目的：当视野中心仅在格子内微调时，可减少服务器请求

1.2 九宫格示意

  • 视野中心点：玩家所在格子
  • 玩家看到的范围：客户端渲染区域（内框）
  • 服务器传输的范围：服务器下发数据的区域（外框，略大于客户端）

1.3 服务端数据结构（Erlang）

二、传统方案的局限

2.1 对象绑定格子（九宫格）

  • 思路：玩家视野覆盖某格子，则将该格子内的对象发给客户端
  • 前提：对象必须严格位于单个九宫格内
  • 问题：若对象超长、跨越多个九宫格，需关联多个格子，移动时从多个格子删除易产生数据错误

2.2 四叉树方案

  • 思路：用树形结构存储对象，支持不同尺寸；大对象可挂在父节点（如 4×4 及以上）
  • 优点：大数据量下检索高效、支持多种对象尺寸、索引快
  • 局限：适合「大块」对象（占满相邻格子），不适合「细长」对象（长而窄）

2.3 超长物体的需求

项目背景：SLG 大地图，X 轴 0–160 格，Y 轴 0–176 格。

新需求：玩家军队从 A 点行军到 B 点，路径可能跨越数百格。策划要求整条路径对沿途所有人可见，且路径因地形障碍呈不规则折线。

朴素做法的问题：将路径直接放入数百个格子：

  • 玩家微调目的地时，需从数百个九宫格中查找并删除旧路径
  • 若有 bug 导致部分格子未删除，路径会碎片化，污染场景
  • 频繁读写大量格子，性能开销大

三、问题抽象

抛开既有实现，从逻辑上重新抽象：

  1. 地图：X 轴 0–160，Y 轴 0–176 的坐标系
  2. 视野：边长为 3 的正方形，部分或全部位于坐标系内
  3. 路径：多条可分段、可多段的折线，用 ID 区分

问题：根据以上条件，求出经过该正方形的路径。

四、解决方案：线段与矩形相交

4.1 问题转换

AOI 由「点是否在范围内」变为：线段（超长物体） 与 小正方形（视野） 是否相交。

  • 更新时只需处理相关线段（如图中 ID 为 3 的线段）
  • 删除时只需移除对应线段，保证数据一致

4.2 排除法（Bounding Box）

因正方形相对较小、地图坐标范围大，采用排除法快速过滤：

设小正方形边界为：X1（最小 X）、X2（最大 X）、Y1（最小 Y）、Y2（最大 Y）。

不相交条件（线段 V 两端点均满足其一即可排除）：

  • 线段两端点 X 均 < X1 → 在正方形左侧
  • 线段两端点 X 均 > X2 → 在正方形右侧
  • 线段两端点 Y 均 < Y1 → 在正方形下方
  • 线段两端点 Y 均 > Y2 → 在正方形上方

可能相交（需进一步精确判断）：

  • 线段任一端点 X 在 [X1, X2] 内，且 Y 在 [Y1, Y2] 内

4.3 性能与实现

  • 上述比较为简单运算，可过滤掉 90% 以上 无关线段
  • 剩余少量线段再做精确的线段-矩形相交判断
  • 逻辑与进程解耦，可多进程并行，性能高
  • 删除操作仅删除对应线段，数据安全

线段与矩形相交的精确算法可在网上查阅，翻译为 Erlang 即可，此处不展开。
