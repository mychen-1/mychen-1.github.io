缓存模块

1.缓存模块

打算以玩家缓存举例。

玩家数据分为三个地方 1.玩家player 基于内存 2.ets缓存 3.mysql（dets）数据库

玩家数据同时只会存在player 或者 ets中。

流程如下：

介绍：

​ 1.第一次查先去dets查到，放入ets中，如果该字段有修改， 放入player，删除ets。

​ 2.落地就是五分钟定时器，刷到MySQL

​ 3.玩家下线 放入ets。

问题点：

1.ets会不会很大， 因为玩家下线是存放到ets中， 确保下次查找快？

​ 答：应该并不会，每次重启的时候ets其实是空的， 对于流失玩家数据不会加载到ets。

​ 优化：①.既然觉得ets大，那每个cache进程起一个定时器（时间自己定）。以玩家为单位，每次更新获取当前时间戳。 当定时器到的时候，时间过久未变化的删除。

​ ②.灵感来源于 erlang内存管理，新生代老年代。对于ets数据引用 计数器，定时器也开启作用于数据 状态变化以及数据清除。

2.既然数据存储于player 内存上，那player结构想必会很大。那方法调用，栈会不会很大效率低下？

项目目前未发现问题。 但是注意点 不要将player结构用于进程传输， 这样会导致进程崩溃的。

3.缺点：

1.所有玩家数据可能存在player或者ets中， 这样导致修数据的时候， 不能拿取整个ets操作。 需要构造玩家进程 再构造player。但是这种构造player其实不大，因为只加载必要数据。 但是还有一个问题，那就是老号的数据也加载进来了。ets有很多无用数据。

2.版本管理

项目设计是一个功能一张表， 并没有使用一张表 player_id, data ，data序列化模式。

跨服 活动 合服总结

浩浩荡荡写这么多， 我觉得精髓是这几个模块之间的联系。 如何高内聚低耦合

思考总结

1.我们跨服是周一进行跨服分组，那跨服活动是跨周的怎么办？

解答：每一个活动有一个group_info字段，这里面记录的cross_group_info结构， 其中有svr_list。 又因为正在开启的活动这些数据不会变化。 所以不会影响本轮活动。

2.活动进程与跨服进程启动顺序是什么？有联系嘛？

顺序 活动进程 -> 跨服进行。

首先获得进程会监听节点连接时间， 跨服活动会去同步。 如果是先开跨服， 跨服抛出的节点连接事件，活动进程可能还没开启，导致数据同步失败。

3.合服的节点 跨服组管理。

合服文章有说，但是设计方式还得理解。

合服

1.合服

合服主要分为两步：

合服前:

  1. cowboy 关闭端口
  2. 踢所有玩家下线
  3. 关活动
  4. 重开某些活动。
  5. 清除老号
  6. 删除cache中老号数据
  7. 关游戏

合服中

  1. 运维修改游戏服配置， 可以取到 合服 和被合服
  2. 将dets文件放入同一台机器， 使用open_file 打开被合服dets
  3. 对应cache有单独处理方法，匹配实现。否则默认

2.注意点

1: 合服中 关活动如何确保活动全部关闭？

ActList是所有活动。
该进程进行阻塞， 对应活动进程会收到消息，某活动处理完毕后会像合服进程发送完毕。

注意点：跨服活动，只有主服才有消息。所以主服才监听跨服活动。

同理： 合服中第三点 cache是否全部处理完 同方法。

2.踢玩家下线，也需要确保嘛？

不需要，正常活动结束的时候，玩家也可能也在线。 我想活动全部关闭了，不可能玩家还没踢下去把。

3.重开某些活动怎么实现的？

只要删除活动对应cache就行。 逻辑：再次开服的时候， 活动进程一定会开启，活动那边就会检测需要重开的活动type，因为cache被删除就会重新加载。

4.上面这么多逻辑，对于跨服组信息好像没处理？ 怎么实现的？

在跨服分组那边节点连接中心节点时， 会携带被合服的组（被合掉的服 不可能再重启了）。 比对新老合服信息。 把多余服进行节点删除。中心服操作（具体可看跨服重组那边）。

5.为什么不合服前的时候处理呢？这样就不需要每次连接校验被合服了。怎么处理？

跨服组信息需要在中心服处理，所以需要运维在中心服设置配置， 合服 -> 被合服。

然后中心服什么时候处理呢？按照旧逻辑 应该是开服的时候。

活动模块

1.数据结构

1.reload_activity 加载活动接口

流程如下：

1.此接口复用高， 进程init使用、跨服组变化、0点事件、等。

1.init

1.监听事件， 0点、配置加载、节点连接、组信息变化。

2.init所有活动。

  1. 先删除本地所有后台活动，再从数据库查询所有后台活动。（实现更新）
  2. 开启所有状态为开启的活动。 根据当前时间设置状态和定时器（只设置当日的， 因为有0点事件）
  3. reload_activity

3.对于跨服活动。1.游戏服是主服，下发到子服。 2.游戏服是子服请求

​ 注意：开服的时候，所有游戏服是无规则开启的。 在第三步同步的时候，很可能其他服未开启，导致同步失败。

​ 解决办法：第一步中 有监听事件节点连接。 节点连接的时候，会去分发或者请求的。

2. group_update 组变化

这个是事件，携带变化的组信息。 重新加载一下活动。 主要是加载未开启，因为已开启的活动数据不会再改变。

未开启活动，检测所有条件，然后检测当前状态，就抛出开启事件。

疑问：对于一个跨服活动，最开始未开启设置开启定时器，然后跨服信息未满足，导致未开启。然后组信息变化。这样活动开启是不是直接跳到活动某个阶段（活动开启也分多种 预备期 展示期）。 这样正常嘛？

总结

1.对于活动来说， 着重点主要是活动开启，状态变化， 0点，和 跨服组变化。

2.注意点是 关服开服时间长， 对于开服时 对于旧活动处理。，

3.单个活动 数据的安全性， 对于跨服活动， 跨服组信息变化。 跨周活动。

跨服分组

1.类型

分为：中心服和游戏服

作用：

​ 1.中心服：负责跨服分组，同步游戏服

​ 2.游戏服：游戏业务玩法。后台对接等。

连接方式：游戏服与中心服互联，游戏服之间也互联。

​ 实现:有一张ets_svr_to_node表，每一个服都存在。由中心服同步。

​ 调用方式：rpc

2.数据结构

3.中心服逻辑

1.游戏服连接(nodeconnect)

  1. 携带nodeinfo信息，主要服务器id，node节点、合服ids等。
  2. 查询该游戏服旧数据，主要是比较新旧合服信息。（由差异 被合服走游戏服删除逻辑）
  3. 遍历所有游戏服类型（聊天组、大组等）， 但是连接只会加入聊天组。
  4. 同步信息到其他游戏服。先同步自己服

2.游戏服退出、删除

接口有游戏服退出和删除。

退出：

1.遍历所有需要退出的组。（从小到大）

2.需要比较组的子分组是什么类型，

​ 2.1.游戏服，组信息直接删除。

​ 2.2.小组等多种游戏服集合的。 寻找该组的父组， 也去删除。

3.通知自己节点更新。

3.每日刷新

  1. 检测游戏服分数， 满足分数的加入组。
  2. 因为节点连接加入了聊天组， 只需要加入 小组 -> 大组
  3. 星期一， 跨服分组
    1. 删除游戏服所有组信息，组信息删除。
    2. 获取所有节点，进行跨服分组。
  4. 检测活动开启
  5. 同步所有信息。

思考：

1.对于组结构中有svr_list字段，是所有节点的id。 在游戏服退出删除的时候， 在组信息中也需要删除，则需要遍历所有该游戏服所有组。

优化1：对于组信息的子分组不是游戏服的时候， 只存储子分组的组id，这样删除退出操作，只需要删除组类型的子分组是游戏服的。

缺点：查询的时候需要递归查询。

2.周一跨服重组的时候，会刷新活动，如果活动是小组、大组并且跨周的时候。怎么处理？

解决：活动那边在活动初始化的时候，存储cross_group_info信息。 跨服组变化的时候 不会改变正在进行的活动。

4. 游戏服逻辑

1.init

1.连接中心服。 携带分数

2.nodedown

1.抛断线事件

2.设置重连定时器

2.sync_node_info（节点信息更新）

1.type = new

1.自己节点连上。设置定时器，更新分数

2.比较新旧组信息

2.type=update

1.主要是更新自己节点组信息

2.比较新旧组信息

3.type=add

有新节点连接，抛事件

2. 存储组信息

单独游戏服 只会存储自己所属组的 组信息。
